# Feature Specification: Keyless generation via the local Claude CLI (US-22)

**Feature Branch**: `022-us22-cli-provider`
**Created**: 2026-07-18
**Status**: Draft
**Input**: User request — add a config-driven mode so that **when no API key is provided**, the app generates answers
through the locally installed **`claude` CLI** (already authenticated on the machine), instead of requiring the user's
own Anthropic API key.

## Context

Today generation requires an Anthropic key: a per-session BYOK key (US-02) or the demo application key (US-03); with
neither, chat is refused (`settings.api_key_missing` 401 / `chat.demo_unavailable` 503). Many developers who run the
app locally already have an **authenticated `claude` CLI** on their machine. US-22 adds a **config flag** that, when
enabled, routes generation through that CLI whenever no key is available — a keyless "it just works" mode for local /
self-hosted runs. It reuses the whole RAG pipeline (retrieval, grounding, citations); only the **generation backend**
changes. The CLI is invoked non-interactively (`claude -p --output-format text`, the grounded prompt on stdin) and
its output streamed back. It is a provider addition behind the existing `IAnswerGenerator` seam (§V), off by default.

## Clarifications

### Session 2026-07-18

- Q: When does the CLI take over? → A: **Only as a keyless fallback** — when the request has no usable key (no BYOK
  key for a normal ask; no application key for a demo ask) **and** the CLI mode is enabled in config. If a key is
  present it is used (unchanged). If the CLI mode is off and no key, the existing key-missing errors stand.
- Q: Runtime requirement? → A: The CLI must be **installed and authenticated on the machine running the API**
  (local `dotnet run` / Aspire, or a self-host image with the CLI). The default container image does not ship the
  CLI; this is documented. Off by default (`ClaudeCli:Enabled=false`).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Chat without a key via the CLI (Priority: P1) 🎯 MVP

With CLI mode enabled and no API key set, the user asks a question and gets a grounded answer generated by the local
`claude` CLI.

**Independent Test**: With `ClaudeCli:Enabled=true` and no key, a chat ask over indexed documents returns a grounded
answer (generation routed to the CLI); the composer is not locked.

**Acceptance Scenarios**:

1. **Given** CLI mode enabled and no BYOK/app key, **When** the user asks a question, **Then** the answer is generated
   by the local `claude` CLI over the retrieved sources (with citations), and no key error is shown.
2. **Given** CLI mode enabled but a BYOK key is set, **When** the user asks, **Then** the BYOK key is used (the CLI is
   not invoked) — the key path is unchanged.

---

### User Story 2 - Graceful when the CLI is unavailable (Priority: P1)

If CLI mode is enabled but the `claude` CLI is missing or fails, the user gets the readable "temporarily unavailable"
message — never a raw error.

**Independent Test**: With CLI mode enabled, no key, and the configured command failing (non-zero / not found), the
ask surfaces `chat.provider_unavailable` (503 / SSE error), not a crash.

**Acceptance Scenarios**:

1. **Given** CLI mode enabled, no key, and the CLI command failing, **When** the user asks, **Then** a readable
   provider-unavailable message is shown (mapped, no stack trace).

---

### Edge Cases

- **CLI mode off, no key**: the existing `settings.api_key_missing` / `chat.demo_unavailable` behaviour is unchanged.
- **Container without the CLI**: with CLI mode on but no `claude` on PATH, an ask degrades to provider-unavailable
  (documented — the CLI must be present on the API host).
- **Prompt safety**: the grounded prompt is passed to the CLI via stdin / an explicit argument list (no shell string
  interpolation), so document/question content cannot inject shell commands.

## Requirements *(mandatory)*

- **FR-001**: A **config flag** (`ClaudeCli:Enabled`, default false) MUST enable CLI-backed generation; all CLI
  settings (command, model, timeout) MUST be configuration-driven (no magic values).
- **FR-002**: When CLI mode is enabled and **no usable key** is available for the request, generation MUST route to
  the local `claude` CLI (non-interactive, the grounded prompt fed safely, not via a shell string).
- **FR-003**: When a usable key **is** available, the existing key path MUST be used unchanged; the CLI MUST NOT be
  invoked.
- **FR-004**: The pre-generation key guard MUST NOT refuse a keyless request when CLI mode is enabled (so the keyless
  chat proceeds); the frontend composer MUST unlock when keyless generation is available.
- **FR-005**: A CLI failure (missing binary, non-zero exit, timeout) MUST map to the existing readable
  provider-unavailable outcome — never a raw error or stack trace.
- **FR-006**: The rest of the pipeline (retrieval, grounding, citations, SSE, session isolation, demo limits) MUST be
  unchanged; the CLI only replaces the generation backend.

## Success Criteria *(mandatory)*

- **SC-001**: With CLI mode on and no key, 100% of asks over indexed documents produce a grounded answer via the CLI
  (given the CLI is installed + authenticated), with the composer unlocked.
- **SC-002**: With a key present, the CLI is invoked in 0% of asks (the key path is used).
- **SC-003**: A failing/missing CLI yields the readable provider-unavailable message in 100% of such cases (no crash).
- **SC-004**: All four test tiers stay green; the change is isolated to the generation provider + one config flag.

## Assumptions

- The `claude` CLI (Claude Code) is installed and authenticated on the API host; the default container does not ship
  it (documented as a local/self-host feature). Off by default.
- The CLI's print mode (`-p --output-format text`) returns the answer text on stdout; token-by-token streaming from
  the CLI is a future enhancement (the answer is delivered as one block for now).
- Session isolation, quota, and demo limits are unaffected (only generation changes).

## Out of Scope

- Shipping the CLI in the production container; token-level streaming from the CLI; a CLI-backed embedding provider.
